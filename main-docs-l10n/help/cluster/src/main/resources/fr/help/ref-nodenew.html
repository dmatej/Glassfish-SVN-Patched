Nouveau noeud

La page Nouveau noeud permet de créer un noeud GlassFish Server. Un noeud représente un hôte sur lequel le logiciel GlassFish Server est installé. Un noeud doit exister au niveau de chaque hôte sur lequel résident des instances GlassFish Server.

Les options présentes sur la page Nouveau noeud dépendent du type de noeud.

Options pour tous les types de noeud

Les options suivantes sont disponibles pour tous les types de noeud :

Nom

Nom du noeud.

Le nom doit respecter les exigences suivantes :

  • Il ne peut contenir que des caractères ASCII.

  • Il doit commencer par une lettre, un chiffre ou un trait de soulignement.

  • Il ne peut contenir que les caractères suivants :

    • Lettres minuscules

    • Lettres majuscules

    • Chiffres

    • Trait d'union

    • Point

    • Trait de soulignement

  • Le nom doit être unique dans le domaine et ne doit pas appartenir à un autre noeud, à une configuration nommée, à un cluster ou à une instance GlassFish Server.

  • Il ne doit pas être domain, server ou tout autre mot-clé réservé par GlassFish Server.

Type

Liste déroulante permettant de spécifier le type de noeud parmi les suivants :

CONFIG

Le noeud ne prend pas en charge la communication à distance. Si CONFIG est sélectionné, aucune option de configuration SSH ou DCOM n'est affichée et le champ Répertoire d'installation est vide.

DCOM

Le noeud prend en charge la communication via le protocole distant DCOM (Distributed Component Object Model). Le protocole DCOM est disponible uniquement pour les systèmes Windows.

SSH

Le noeud prend en charge la communication via le protocole SSH (shell sécurisé).

Hôte du noeud

Nom de l'hôte que le noeud représente. Le caractère obligatoire de ce champ dépend du type du noeud :

  • Si le type du noeud est CONFIG, le nom de l'hôte peut être omis. Le nom de l'hôte peut être déterminé lors de la création des instances résidant sur le noeud.

  • Si le type du noeud est DCOM ou SSH, le nom de l'hôte doit être spécifié. Sinon, une erreur se produit.

Répertoire du noeud

Chemin d'accès complet au répertoire devant contenir les instances GlassFish Server créées sur le noeud. Si un chemin relatif est indiqué, il est relatif au répertoire as-install.

La valeur par défaut dépend du type du noeud :

  • Si le type du noeud est CONFIG, aucun répertoire d'instances n'est spécifié pour le noeud dans la configuration du serveur d'administration de domaine (DAS). Le répertoire du noeud peut être déterminé lors de la création des instances résidant sur le noeud.

  • Si le type du noeud est DCOM, la valeur par défaut est as-install\nodes, où as-install correspond au répertoire d'installation de base du logiciel GlassFish Server sur l'hôte.

  • Si le type du noeud est SSH, la valeur par défaut est as-install/nodes, où as-install correspond au répertoire d'installation de base du logiciel GlassFish Server sur l'hôte.

Répertoire d'installation

Chemin d'accès complet au parent du répertoire d'installation de base du logiciel GlassFish Server sur l'hôte (par exemple, /export/glassfish3).

La valeur par défaut dépend du type du noeud :

  • Si le type du noeud est CONFIG, aucun répertoire d'installation n'est spécifié pour le noeud dans la configuration du serveur DAS. Le répertoire d'installation peut être déterminé lors de la création des instances résidant sur le noeud.

  • Si le type du noeud est DCOM ou SSH, la valeur par défaut est le parent du répertoire d'installation de base par défaut du logiciel GlassFish Server pour le serveur d'administration de domaine (DAS). Ce comportement par défaut n'est utile que si vous installez GlassFish Server au même emplacement sur tous les hôtes.

Options supplémentaires pour les noeuds DCOM

Les options suivantes sont disponibles uniquement si le champ Type a la valeur DCOM :

Installer GlassFish Server

Si l'option Activé est sélectionnée, le serveur DAS copie le logiciel GlassFish Server à partir de l'hôte de serveur DAS vers l'hôte du noeud. Pour copier le logiciel GlassFish Server vers l'hôte, le serveur DAS doit être capable de contacter l'hôte via DCOM.

Forcer

Si l'option Activé est sélectionnée, le noeud est créé dans la configuration du serveur DAS même si la validation des paramètres du noeud échoue. Pour valider les paramètres d'un noeud, le serveur DAS doit être capable de contacter l'hôte du noeud via DCOM.

Valider DCOM

Si l'option Activé est sélectionnée, GlassFish Server teste la connexion via DCOM à l'hôte du noeud avant de créer le noeud. Dans le cadre du test, GlassFish Server copie un script nommé delete_me.bat vers l'hôte.

Si vous sélectionnez cette option, le champ Répertoire de test distant est activé pour vous permettre d'indiquer le répertoire dans lequel GlassFish Server doit copier le script.

Répertoire de test distant

Dossier de l'hôte distant dans lequel GlassFish Server doit créer le fichier delete_me.bat. L'utilisateur indiqué par l'option Nom utilisateur Windows doit disposer d'un accès en écriture à ce dossier. Sinon, une erreur se produit. La valeur par défaut est C:\.

Domaine Windows

Nom du domaine Windows contenant l'utilisateur indiqué par l'option Nom utilisateur Windows. La valeur par défaut est le nom de l'hôte sur lequel la Console d'administration est en cours d'exécution.

Nom utilisateur Windows

Utilisateur sur l'hôte de ce noeud qui doit exécuter le processus de connexion à l'hôte via DCOM. La valeur par défaut est l'utilisateur qui exécute le processus DAS. Si la valeur de l'option Hôte du noeud est localhost-domain, l'option Nom utilisateur Windows n'est pas prise en compte.

Authentification de l'utilisateur Windows

Liste déroulante permettant d'indiquer la manière dont l'utilisateur Windows est authentifié lors de la connexion à l'hôte de ce noeud. Les méthodes possibles pour l'authentification sont les suivantes :

Mot de passe

L'utilisateur Windows est authentifié par le biais d'un mot de passe stocké dans la configuration du serveur DAS.

Attention:

Ce mot de passe est stocké sous forme de texte en clair dans la configuration du serveur DAS. Pour une sécurité accrue, utilisez un alias de mot de passe.

Si vous sélectionnez cette option, le champ Mot de passe utilisateur Windows est activé pour vous permettre d'indiquer le mot de passe.

Alias de mot de passe

L'utilisateur Windows est authentifié par le biais d'un alias de mot de passe. Si vous sélectionnez cette option, choisissez l'alias de mot de passe dans la liste déroulante correspondante.

Mot de passe utilisateur Windows

Mot de passe que l'utilisateur Windows doit utiliser lors de la connexion à l'hôte de ce noeud.

Alias de mot de passe

Liste déroulante des alias de mot de passe définis pour ce domaine.

Options supplémentaires pour les noeuds SSH

Les options suivantes sont disponibles uniquement si le champ Type a la valeur SSH :

Installer GlassFish Server

Si l'option Activé est sélectionnée, le serveur DAS copie le logiciel GlassFish Server à partir de l'hôte de serveur DAS vers l'hôte du noeud. Pour copier le logiciel GlassFish Server vers l'hôte, le serveur DAS doit être capable de contacter l'hôte via SSH.

Forcer

Si l'option Activé est sélectionnée, le noeud est créé dans la configuration du serveur DAS même si la validation des paramètres du noeud échoue. Pour valider les paramètres d'un noeud, le serveur DAS doit être capable de contacter l'hôte du noeud via SSH.

Port SSH

Port à utiliser pour les connexions SSH à l'hôte de ce noeud. La valeur par défaut est 22. Si la valeur de l'option Hôte du noeud est localhost-domain, l'option Port SSH n'est pas prise en compte.

Nom utilisateur SSH

Utilisateur sur l'hôte de ce noeud qui doit exécuter le processus de connexion à l'hôte via SSH. La valeur par défaut est l'utilisateur qui exécute le processus DAS. Pour garantir que le serveur DAS puisse lire le fichier de clés privées SSH de l'utilisateur, indiquez l'utilisateur qui exécute le processus DAS. Si la valeur de l'option Hôte du noeud est localhost-domain, l'option Nom utilisateur n'est pas prise en compte.

Authentification de l'utilisateur SSH

Liste déroulante permettant d'indiquer la manière dont l'utilisateur SSH est authentifié lors de la connexion à l'hôte de ce noeud. Les méthodes possibles pour l'authentification sont les suivantes :

Fichier de clés

L'utilisateur SSH est authentifié via un fichier de clés privées SSH. Si vous sélectionnez cette option, spécifiez le chemin d'accès absolu au fichier dans le champ Fichier de clés.

Mot de passe

L'utilisateur SSH est authentifié via un mot de passe stocké dans la configuration du serveur DAS.

Attention:

Ce mot de passe est stocké sous forme de texte en clair dans la configuration du serveur DAS. Pour une sécurité accrue, utilisez un fichier de clés ou un alias de mot de passe.

Si vous sélectionnez cette option, le champ Mot de passe utilisateur SSH est activé pour vous permettre d'indiquer le mot de passe.

Alias de mot de passe

L'utilisateur SSH est authentifié via un alias de mot de passe. Si vous sélectionnez cette option, choisissez l'alias de mot de passe dans la liste déroulante correspondante.

Configurer SSH

Si l'option Activé est sélectionnée, GlassFish Server configure une clé SSH sur l'hôte du noeud. GlassFish Server configure la clé SSH en copiant les fichiers de clés existants à partir de l'hôte de serveur DAS ou, si nécessaire, en générant les fichiers avant de les copier. Cette option est disponible uniquement si l'option Authentification de l'utilisateur SSH a la valeur Fichier de clés.

Générer une clé

Si l'option Activé est sélectionnée, GlassFish Server génère les fichiers de clés SSH avant de les copier sur l'hôte du noeud si les fichiers n'existent pas. Si les fichiers de clés SSH existent, GlassFish Server ne génère pas les fichiers, même si l'option Activé est sélectionnée.

Remarque:

Si les fichiers de clés SSH n'existent pas, vous devez sélectionner l'option Activé. Sinon, une erreur se produit lorsque vous tentez d'enregistrer les modifications.
Mot de passe utilisateur SSH

Mot de passe que l'utilisateur SSH doit utiliser lors de la connexion à l'hôte de ce noeud.

Alias de mot de passe

Liste déroulante des alias de mot de passe définis pour ce domaine.

Fichier de clés publiques

Chemin d'accès absolu au fichier de clés publiques SSH spécifié par l'option Nom utilisateur SSH. Le contenu du fichier de clés publiques est ajouté à la fin du fichier .ssh/authorized_keys de l'utilisateur sur l'hôte du noeud. Si le fichier .ssh/authorized_keys n'existe pas sur l'hôte, GlassFish Server le crée.

Fichier de clés

Chemin d'accès absolu au fichier de clés privées SSH spécifié par l'option Nom utilisateur SSH. Ce fichier est utilisé pour l'authentification auprès du démon sshd sur l'hôte du noeud.

Remarque:

GlassFish Server prend également en charge l'authentification par mot de passe via l'entrée AS_ADMIN_SSHPASSWORD dans le fichier de mots de passe. Le fichier de mots de passe est spécifié dans l'option -paswordfile de l'utilitaire asadmin(1).

Si le fichier de clés privées SSH est protégé par une phrase secrète, le fichier de mots de passe doit contenir l'entrée AS_ADMIN_SSHKEYPASSPHRASE.

Le serveur DAS doit disposer de droits d'accès et de lecture sur le fichier de clés.

La valeur par défaut est un fichier de clés placé dans le répertoire .ssh de l'utilisateur. Si plusieurs fichiers de clés sont disponibles, GlassFish Server utilise l'ordre de préférence suivant :

  1. id_rsa

  2. id_dsa

  3. identity
