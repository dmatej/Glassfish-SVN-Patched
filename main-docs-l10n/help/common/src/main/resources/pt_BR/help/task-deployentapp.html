Para Implantar uma Aplicação Enterprise

  1. Na árvore de navegação, selecione o nó Aplicações.

    A página Aplicações será aberta.

  2. Na tabela Aplicações Implantadas, clique no botão Implantar.

    A página Implantar Aplicações ou Módulos será exibida.

  3. Especifique o local da aplicação a ser implantada.

    • Se a aplicação for um arquivo compactado que reside na máquina cliente ou que pode ser acessado a partir dela, selecione a opção Arquivo Encapsulado Cujo Upload Será Feito para o Servidor.

      O computador cliente é o host em que você exibe o Console de Administração por meio de um browser.

      Clique em Procurar para localizar o arquivo ou digite o caminho completo para o arquivo.

    • Se a aplicação for um arquivo compactado que reside na máquina servidora ou se for uma aplicação não encapsulada de um diretório explodido, selecione a opção Arquivo Encapsulado ou Diretório Local Acessível no GlassFish Server.

      O computador servidor é o host que executa o servidor de administração de domínios (DAS).

      Clique em Procurar Arquivos para localizar um arquivo ou clique em Procurar Pastas para localizar um diretório. Como alternativa, digite o caminho completo para o arquivo ou diretório. A implantação com base em um diretório explodido destina-se a desenvolvedores avançados e não é recomendada para ambientes de produção.

  4. Na lista drop-down Tipo, selecione a Aplicação Enterprise.

    Se você tiver especificado um Arquivo Encapsulado Cujo Upload Será Feito para o Servidor ou um Arquivo Encapsulado Local Acessível no GlassFish Server, o tipo da aplicação será detectado automaticamente. Se você tiver especificado um Diretório Acessado do GlassFish Server, você deve especificar o tipo manualmente.

  5. No campo Nome da Aplicação, preserve o nome default, que é o nome do diretório ou o prefixo do nome do arquivo, ou digite outro nome.

    O nome default será exibido se você optar por fazer upload de um arquivo. O nome da aplicação deve ser exclusivo.

    O nome pode incluir um identificador de versão opcional, que segue o nome e é separado do nome por dois-pontos (:). O identificador de versão deve começar com uma letra ou um número. Ela pode conter caracteres alfanuméricos e sublinhado (_), traço (-), e ponto (.). Para obter mais informações sobre versões modulares ou da aplicação, consulte Module and Application Versions em Oracle GlassFish Server Application Deployment Guide.

  6. Na lista drop-down Servidores Virtuais, selecione o servidor virtual a ser associado à aplicação.

    A opção Servidores Virtuais aparece somente se a instância default do servidor, server, existir. Se os clusters ou outras instâncias de servidores standalone existirem, é possível selecionar os servidores virtuais depois da implantação. Vá à página Editar Aplicação, selecione a guia Alvo e selecione Gerenciar Servidores Virtuais para o alvo desejado.

  7. Marque a caixa de seleção Status Ativado para ativar a aplicação.

    Por default, essa opção está selecionada. Esta opção estiver desativada, a aplicação não estará disponível.

  8. Marque a caixa de seleção Disponibilidade Ativada para ativar a persistência de sessão e o checkpoint SFSB da aplicação.

    Se a caixa de seleção Ativado estiver selecionada, a alta disponibilidade será ativada para sessões Web e para checkpoint e, potencialmente, a passivação do bean da sessão stateful (SFSB). Se esta opção estiver definida como falsa (default), todos os salvamentos de sessões Web e checkpoints SFSB serão desativados para a aplicação especificada. Se definida como verdadeira, a aplicação especificado será ativada para alta disponibilidade. Defina essa opção como verdadeira apenas se a alta disponibilidade estiver configurada e ativada em níveis mais altos, como os níveis do servidor e contêiner.

    Esta opção aparece se clusters ou instâncias de servidores standalone, além da instância default do servidor (server), existirem.

  9. Marque a caixa de seleção Java Web Start para ativar o acesso ao Java Web Start para um cliente de aplicação.

    Por default, a opção está ativada.

  10. Marque a caixa de seleção Precompilar JSPs para precompilar arquivos JSP (JavaServer Pages).

    Se esta opção for desativada, os arquivos JSP serão compilados durante o runtime, quando forem acessados pela primeira vez. Por default, essa opção está desativada.

  11. Marque a caixa de seleção Executar Verificador para verificar os descritores de implantação antes da implantação.

    Se essa opção for selecionada e a verificação falhar, a implantação não será realizada. O verificador examina a estrutura e o conteúdo dos descritores de implantação. A verificação de aplicações grandes geralmente consome muito tempo. Por default, essa opção está desativada.

    Os pacotes do verificador devem ser instalados a partir da Ferramenta de Atualização, ou uma advertência será registrada e esta opção será ignorada.

  12. Marque a caixa de seleção Compatibilidade para utilizar os requisitos de visibilidade JAR do GlassFish Server v2 para aplicações em lugar dos estritos requisitos Java EE 6 implantados nas versões do GlassFish Server v3, incluindo 4.0.

    Por default, essa opção está desativada.

    A especificação da plataforma Java EE 6 impõe exigências mais estritas do que o Java EE 5, no qual os arquivos JAR podem estar visíveis a vários módulos em um arquivo EAR. Especificamente, os clientes da aplicação não devem ter acesso a arquivos EJB JAR ou outros arquivos JAR no arquivo EAR, a menos que as referências utilizem os mecanismos EE Java padrão (extensões, por exemplo) ou o mecanismo biblioteca-diretório Java EE. Se esta opção for marcada, estas restrições do Java EE 6 são removidas.

  13. Marque a caixa de seleção Forçar Reimplantação para forçar a reimplantação, caso a aplicação já esteja implantada.

    Se esta opção não tiver sido selecionada, a tentativa de implantar uma aplicação já implantada resultará em erro. Por default, essa opção está desativada.

  14. Marque a caixa de seleção Manter Estado para conservar as sessões Web, as instâncias SFSB e os timers EJB criados persistentemente entre as reimplantações.

    Por default, essa opção está desativada. Esta opção somente é suportada na instância do servidor default, com o nome server. Não é suportada e ignorada em nenhum outro alvo.

    Algumas alterações em uma aplicação entre as reimplantações impedem que este recurso funcione apropriadamente. Por exemplo, não altere o conjunto de variáveis da instância na classe do bean SFSB.

    Para aplicações Web, este recurso somente é aplicável se no arquivo glassfish-web-app.xml o atributo persistence-type do elemento session-manager for file.

    Em instâncias do bean de sessão stateful, o tipo de persistência sem alta disponibilidade está definido no servidor (a opção Tipo de persistência SFSB) e deve estar definido como file, que é o valor default e recomendado.

    Se qualquer sessão Web ativa, instância SFSB ou timer EJB falhar em ser preservado ou restaurado, nenhum deles estará disponível quando a reimplantação tiver sido concluída. Entretanto, a reimplantação continuará e uma advertência será registrada em log.

    Para preservar dados do estado ativo, o GlassFish Server serializa os dados e os salva na memória. Para restaurar os dados, o carregador de classe da aplicação recém-reimplantada desserializa todos os dados salvos anteriormente.

  15. No campo Ordem de Implantação, insira um número para especificar a ordem de implantação da aplicação.

    Aplicações com um número mais baixo são carregadas primeiro na inicialização do servidor. Um aplicação com uma ordem de implantação de 102 é carregada antes de uma de 110. Se uma ordem de implantação não for especificada no tempo uma aplicação será implantada, a ordem de implantação default de 100 será designada. Se duas aplicações tiverem a mesma ordem de implantação, a aplicação que foi implantada primeiro será carregada primeiro. Especificar uma ordem de implantação é útil se a aplicação tiver dependências e deve ser carregada em uma determinada ordem.

  16. No campo Bibliotecas, digite uma lista separada por vírgulas dos arquivos JAR da biblioteca, específicos deste módulo ou aplicação.

    Os caminhos podem ser absolutos ou relativos. Um caminho relativo a domain-dir/lib/applibs. Se o caminho for absoluto, o caminho deve ficar acessível para o servidor de administração de domínio (DAS) que significa que ele deve estar sob domain-dir. As bibliotecas são disponibilizadas para a aplicação na ordem em que elas são especificadas.

  17. No campo Descrição, digite uma descrição para esta aplicação.

  18. Para adicionar instâncias de servidor e clusters do alvo nos quais implantar o módulo ou aplicação, selecione-os na lista Alvos Disponíveis e utilize o botão Adicionar para movê-los para a lista Alvos Selecionados.

    A opção Alvo aparece se clusters ou instâncias de servidores standalone, além da instância default do servidor (server), existirem. Por default, o módulo ou aplicação é implantado na instância default do servidor, denominado server. Utilize a botão Adicionar Tudo para implantar em todos os alvos. Utilize os botões Remover e Remover Tudo para remover os alvos da lista Alvos Selecionados.

  19. Clique em OK.
