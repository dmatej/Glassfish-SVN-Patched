アプリケーション構成

「アプリケーション構成」ページを使用して、デプロイされたアプリケーションに対する変更の検出と変更されたクラスの再ロードを確実に実行できるプロパティを設定します。

「アプリケーション構成」ページには、次のトピックが含まれます。

デフォルトのロード

デフォルト値のある設定をデフォルト値に復元するボタン。デフォルト値のない設定は変更されません。

再ロード

このオプションが有効な場合、変更されたクラスは動的に再ロードされます。サーバーは、デプロイメント・ディレクトリ内の変更を定期的に確認し、変更に伴いアプリケーションを自動的および動的に再デプロイします。この機能は、変更したコードをすぐにテストできるため、開発環境で役に立ちます。しかし、本番環境では、動的再ロードはパフォーマンスを低下させる可能性があります。また、再ロードが行われているときは、その転送時のセッションが無効になります。クライアントのセッションを再起動する必要があります。このオプションはデフォルトで有効になっています。

再ロードのポーリング間隔

デプロイされたアプリケーションとモジュール内のコード変更をサーバーがチェックする頻度を識別します。デフォルト値は2です。

管理セッション・タイムアウト

管理コンソールがタイムアウトし、再度ログインが必要となるまでの非アクティブ状態の分数を識別します。デフォルト値は60です。

自動デプロイ

このオプションが有効な場合、自動デプロイ・ディレクトリ内のアプリケーションは自動的にデプロイされます。このオプションはデフォルトで有効になっています。

自動デプロイのポーリング間隔

アプリケーションまたはモジュールの変更用の自動デプロイ・ディレクトリをサーバーがチェックする頻度を指定します。ポーリング間隔を変更しても、アプリケーションやモジュールのデプロイにかかる時間には影響ありません。デフォルト値は2です。

自動デプロイの再試行タイムアウト

エラーが発生するまでに部分的にコピーされたファイルのサイズが変更されずに残る秒数を指定します。デフォルト値は4です。

自動デプロイ・ディレクトリへのファイルのコピーに時間がかかると、ファイル全体がコピーされる前にそのファイルが表示されることがあります。その結果、アプリケーションの自動デプロイが失敗します。この理由によってアプリケーションの自動デプロイに失敗した場合、GlassFish Serverはアプリケーションのデプロイを再試行します。部分的にコピーされたファイルのサイズが、指定された時間を過ぎても変更されないと、エラーが発生します。

自動デプロイ・ディレクトリ

アプリケーションの自動デプロイメントをモニターするディレクトリを示します。デフォルト値はdomain-dir/autodeployです。

XML検査

標準およびGlassFish Serverのデプロイメント記述子で実行される、XML検査のタイプを指定します。次のいずれかのタイプを指定できます。

完全

XML検査が失敗すると、デプロイメントも失敗します。この値はデフォルトです。

解析中

XML検査のエラーがレポートされますが、デプロイメントは実行されます。

なし

XML検査は実行されません。

ベリファイア

このオプションが有効な場合は、自動デプロイメントの前にベリファイアが実行されます。確認が失敗すると、デプロイメントは実行されません。ベリファイアはファイルの構造とコンテンツを調べます。大きなアプリケーションの確認は時間がかかる可能性があります。このオプションは、デフォルトでは無効になっています。

プリコンパイル

このオプションが有効な場合は、JavaServer Pages (JSP)ファイルがプリコンパイルされます。このオプションが無効な場合、JSPファイルは初めてアクセスされた実行時にコンパイルされます。このオプションは、デフォルトでは無効になっています。

追加プロパティ

アプリケーションの追加プロパティ。GlassFish Serverはアプリケーションの追加プロパティを定義しません。
