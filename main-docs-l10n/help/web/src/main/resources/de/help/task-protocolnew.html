So wird ein Protokoll erstellt

  1. Blenden Sie in der Navigationsbaumstruktur den Knoten "Konfiguration" ein.

  2. Blenden Sie unter dem Konfigurationsknoten den Knoten "Netzwerkkonfiguration" ein.

  3. Wählen Sie unter dem Knoten "Netzwerkkonfiguration" den Knoten "Protokolle".

    Die Seite "Protokolle" wird geöffnet.

  4. Klicken Sie auf der Seite "Protokolle" auf "Neu".

    Die Seite "Neues Protokoll" wird geöffnet.

  5. Geben Sie im Feld "Name" einen eindeutigen Namen für das Protokoll ein.

  6. Wählen Sie das Kontrollkästchen "Sicherheit aktiviert", um Sicherheit für das Protokoll zu aktivieren.

    Diese Option ist standardmäßig deaktiviert. Wenn Sie Sicherheit aktivieren, können Sie die Sicherheitseinstellungen auf der Registerkarte "SSL" der Seite "Protokoll bearbeiten" ändern.

  7. Wählen Sie das Kontrollkästchen "Status aktiviert", um den Dateicache zu aktivieren.

    Diese Option ist standardmäßig aktiviert.

  8. Geben Sie im Feld "Maximales Alter" das maximale Alter (in Sekunden) für einen gültigen Cacheeintrag ein.

    Dieser Parameter kontrolliert, wie lange gecachte Informationen verwendet werden, nachdem eine Datei gecacht wurde. Ein Eintrag, der älter ist als das angegebene Maximalalter, wird durch einen neuen Eintrag für dieselbe Datei ersetzt. Wenn sich der verwendete Content nur selten ändert, erhöhen Sie diesen Wert, um die Performance zu verbessern.

    Das optimale maximale Alter hängt davon ab, ob die vorhandenen Dateien regelmäßig geändert werden. Wenn die Dateien beispielsweise vier Mal täglich in regelmäßigen Abständen geändert werden, können Sie das maximale Alter auf 21600 Sekunden (6 Stunden) festlegen. Andernfalls sollten Sie das maximale Alter auf den längsten Zeitraum einstellen, für den die frühere Version einer Inhaltsdatei auf dem Server verbleiben soll, nachdem die Datei geändert wurde.

    Der Standardwert ist 30.

  9. Geben Sie im Feld "Maximale Cachegröße" die Maximalgröße (in Byte) des Dateicache auf dem Datenträger an.

    Der Standardwert ist 10485760.

  10. Geben Sie im Feld "Maximale Anzahl an Dateien" die maximale Anzahl der Dateien an, die im Dateicache gespeichert werden können.

    Wenn dieser Wert zu hoch gewählt ist, cacht der Server selten verwendete Dateien, wodurch Arbeitsspeicher vergeudet wird. Ist dieser Wert zu niedrig eingestellt, gehen die Vorteile der Cachings verloren. Probieren Sie verschiedene Werte für dieses Attribut aus, um die optimale Einstellung für spezifische Anwendungen zu finden.

    Der Standardwert ist 1024.

  11. Geben Sie im Feld "Servername" den Hostnamen ein, der in den URLs verwendet wird, die der Server an den Client sendet.

    Dieser Name ist der Aliasname, falls Ihr Server einen Alias verwendet. Wenn Ihr Server keinen Alias verwendet, lassen Sie dieses Feld leer.

    Dieser Wert wirkt sich auf URLs aus, die vom Server automatisch generiert werden; URLs für Verzeichnisse und im Server gespeicherte Dateien sind davon nicht betroffen. Wenn Ihr Server einen Alias verwendet, muss der Servername dem Aliasnamen entsprechen. Werden ein Doppelpunkt und eine Portnummer angehängt, wird dieser Port in URLs verwendet, die der Server an den Client sendet.

  12. Wählen Sie aus der Dropdown-Liste "Standard-Virtual Server" den virtuellen Server, der mit dem Protokoll verknüpft werden soll.

    Auf der Seite "Virtuelle Server" können Sie virtuelle Server definieren.

  13. Geben Sie in das Feld "Portumleitung" den Wert des Ports ein, der bei der Umleitung einer Anforderung an einen anderen Port verwendet werden soll.

    GlassFish Server leitet Anforderungen automatisch weiter, wenn die folgenden zwei Bedingungen erfüllt sind:

    • Der Netzwerk-Listener, der auf dieses Protokoll verweist, unterstützt Nicht-SSL-Anforderungen.

    • Es wird eine Anforderung empfangen, für die ein übereinstimmendes Sicherheits-Constraint einen SSL-Transport erfordert.

    Wird kein Umleitungsport angegeben, verwendet GlassFish Server die in der ursprünglichen Anforderung angegebene Portnummer. Standardmäßig ist kein Port angegeben.

  14. Geben Sie im Feld "Maximale Verbindungen" die maximale Anzahl an Anforderungen an, die in die Pipeline gestellt werden können, bis die Verbindung vom Server geschlossen wird.

    Das Keep-Alive-Teilsystem ruft regelmäßig Verbindungen im Leerlaufstatus ab. Der Standardwert ist 256.

    Setzen Sie diese Eigenschaft auf 1, um HTTP/1.0 Keep-Alive sowie HTTP/1.1 Keep-Alive und Pipelining zu deaktivieren. Der Wert 0 bedeutet, dass Anforderungen immer zurückgewiesen werden. Bei einem Wert -1 ist die Anzahl der Keep-Alive-Verbindungen unbegrenzt.

  15. Geben Sie im Feld "Timeout" die maximale Anzahl Sekunden an, die eine Verbindung als im Leerlauf befindlich betrachtet und im Keep-Alive-Zustand gehalten werden kann.

    Ein Wert kleiner oder gleich null (0) bedeutet, dass Keep-Alive-Verbindungen unbegrenzt offen gehalten werden. Der Standardwert ist 30 Sekunden.

  16. Aktivieren Sie das Kontrollkästchen "Timeout beim Hochladen", um die Verbindung für ein Servlet, das Byte langsam liest, nach Erreichen des Grenzwerts für das Timeout beim Hochladen der Verbindung zu schließen.

    Bei deaktivierter Option tritt für Servlet-Verbindungen kein Timeout ein. Diese Option ist standardmäßig deaktiviert.

  17. Geben Sie im Feld "Timeout beim Hochladen der Verbindung" das Timeout beim Hochladen in Millisekunden an.

    Dieses Feld steht nur zur Verfügung, wenn das Kontrollkästchen "Timeout beim Hochladen" aktiviert ist. Der Standardwert ist 300.000 Millisekunden.

  18. Geben Sie im Feld "Timeout für Anforderung" die Anzahl Sekunden ein, bevor die Anforderung wegen Timeout abgebrochen wird.

    Wenn die Anforderung nicht vor Erreichen des Timeoutwerts verarbeitet wird, wird sie ignoriert. Der Standardwert ist 30 Sekunden.

  19. Geben Sie im Feld "Größe des Sendepuffers" die Größe des Sendepuffers in Byte an.

    Der Standardwert ist 8192 Byte.

  20. Geben Sie im Feld "Headerpuffergröße" die Größe (in Byte) des Puffers an, der von den Anforderungsverarbeitungsthreads zum Lesen der Anforderungsdaten verwendet wird.

    Der Standardwert ist 8192 Byte.

  21. Geben Sie im Feld "Maximale Post-Größe" die maximale Größe von POST-Aktionen in Byte an.

    Der Standardwert ist 2.097.152 Byte.

  22. Geben Sie im Feld "URI-Codierung" den Namen des Zeichensatzes ein, der zum Decodieren der erhaltenen Anforderungs-URIs verwendet wird.

    Als Wert muss ein gültiger IANA-Zeichensatzname angegeben werden. Der Standardwert ist UTF-8.

  23. Geben Sie im Feld "Version" die Version des verwendeten HTTP-Protokolls an.

    Der Standardwert ist HTTP/1.1.

  24. Wählen Sie aus der Dropdown-Liste "Komprimierung" die HTTP/1.1 GZIP-Komprimierung, um Serverbandbreite zu sparen.

    Zur Auswahl stehen:

    On

    Komprimiert Daten.

    Off

    In der folgenden Tabelle werden diese Eigenschaften aufgelistet.

    force

    Erzwingt die Datenkomprimierung in allen Fällen.

    Der Standardwert ist "Off".

  25. Geben Sie im Feld "Komprimierbare MIME-Typen" eine durch Komma getrennte Liste von MIME-Typen ein, für die HTTP-Komprimierung verwendet wird.

    Der Standardwert ist text/html,text/xml,text/plain.

  26. Geben Sie im Feld "Mindestkomprimierungsgröße" die minimale Größe einer komprimierten Datei ein.

    Dieser Wert muss festgelegt werden, wenn das Kontrollkästchen "Komprimierung" aktiviert wird. Der Standardwert ist 2048 Byte.

  27. Geben Sie im Feld "User Agent ohne Komprimierung" eine durch Komma getrennte Liste von regulären Ausdrücken ein, die User Agents von HTTP-Clients entsprechen, für die keine Komprimierung verwendet werden soll.

    Standardmäßig handelt es sich bei diesem Wert um eine leere Zeichenfolge.

  28. Geben Sie im Feld "Beschränkte User Agents" eine Liste von beschränkten User Agents ein, auf die HTTP-Komprimierung angewendet wird.

    Wenn keine User Agents angegeben werden, wird die HTTP-Komprimierung auf alle User Agents angewendet. Standardmäßig sind keine User Agents angegeben.

  29. Geben Sie im Feld "Standardmäßiger Antworttyp" eine Zeichenfolge ein, die den standardmäßigen Antworttyp angibt.

    Das Format ist eine durch Semikolon getrennte Zeichenfolge bestehend aus Contenttyp, Codierung, Sprache und Zeichensatz. Der Standardwert ist text/html; charset=iso-8859-1.

  30. Geben Sie im Feld "Erzwungener Antworttyp" eine Zeichenfolge ein, die den zu verwendenden Antworttyp angibt, falls keine MIME-Zuordnung mit der Dateierweiterung übereinstimmt.

    Das Format ist eine durch Semikolon getrennte Zeichenfolge bestehend aus Contenttyp, Codierung, Sprache und Zeichensatz. Der Standardwert ist text/html; charset=iso-8859-1.

  31. Geben Sie im Feld "Adapter" den Klassennamen des statischen Ressourcenadapters ein.

    Der Standardwert ist com.sun.grizzly.tcp.StaticResourcesAdapter.

  32. Aktivieren Sie das Kontrollkästchen "Comet-Unterstützung", um Comet-Unterstützung für den Netzwerk-Listener zu aktivieren, der auf dieses Protokoll verweist.

    Standardmäßig ist diese Option deaktiviert. Wenn Ihr Servlet oder Ihre JSP-Seite Comet-Technologie verwendet, stellen Sie sicher, dass es bzw. sie beim Starten von GlassFish Server initialisiert wird, indem Sie das Element load-on-startup zur Datei web.xml hinzufügen. Beispiel:

    <servlet>
<servlet-name>CheckIn</servlet-name>
<servlet-class>CheckInServlet</servlet-class>
<load-on-startup>0</load-on-startup>
</servlet>

  33. Aktivieren Sie das Kontrollkästchen "DNS-Suche", um die DNS-Suche zu aktivieren.

    Diese Option ist standardmäßig deaktiviert.

  34. Aktivieren Sie das Kontrollkästchen "RCM-Unterstützung", um Resource Configuration Management (RCM) zu aktivieren.

    Diese Option ist standardmäßig deaktiviert.

  35. Aktivieren Sie das Kontrollkästchen "Trace", um den TRACE-Vorgang zu aktivieren.

    Bei deaktivierter Option ist GlassFish Server weniger anfällig für siteübergreifende Skriptangriffe. Diese Option ist standardmäßig aktiviert.

  36. Wählen Sie das Kontrollkästchen "Autorisierungsdurchlauf aktiviert", um anzugeben, dass der Netzwerk-Listener, der dieses Protokoll verwendet, Daten von einem SSL-beendenden Proxyserver empfängt.

    Diese Option ist standardmäßig deaktiviert.

  37. Aktivieren Sie das Kontrollkästchen "Erstellen von Datenblöcken", um das Erstellen von Datenblöcken für HTTP-Antworten zu aktivieren.

    Diese Option ist standardmäßig aktiviert.

  38. Aktivieren Sie das Kontrollkästchen "XPowered-By", um GlassFish Server anzuweisen, X-Powered-By -Header entsprechend den Spezifikationen für Java Servlet 3.0 und Java Server Pages 2.0 zu verwenden.

    Diese Option ist standardmäßig aktiviert.

Nächste Schritte

Nach der Erstellung eines Protokolls können Sie einen Netzwerk-Listener erstellen, der dieses Protokoll verwendet.
